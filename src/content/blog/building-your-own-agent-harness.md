## 一个公式

如果你对 Agent 智能体的理解只停留在"ChatGPT 能聊天"，那这篇文章会刷新你的认知。

先记住这个公式：

> **Agent = LLM + Harness**

LLM 大语言模型相当于大脑。Harness 是它的躯体、手脚和百宝箱。

裸模型只会说话。加上 Harness，它才能做事。这就是 Agent。

![Agent = LLM + Harness](/images/building-your-own-agent-harness.png)

这不是比喻。这是 2025 年以来整个 AI Agent 领域的工程共识。Claude Code 如此，OpenClaw 如此，Hermes Agent 如此——它们的差异不在模型，在 Harness。

## 你其实已经在用 Harness 了

很多人没意识到，Claude Code 本身就是一个 Agent Harness 的标杆实现。

当你打开 Claude Code，在终端里输入一句自然语言指令，它做了什么？它读你的 CLAUDE.md 了解项目背景，搜索你的文件结构理解当前状态，调用自己的记忆系统回忆上次对话的上下文，需要时启动子代理并行处理。这不是"聊天"，这是一个完整的智能行动循环。

Claude Code 的核心架构极其简洁：**模型 + 工具循环 + 上下文管理。**

一个循环里，模型思考、调用工具、观察结果、继续思考，直到任务完成。工具包括文件读写、终端执行、代码搜索。上下文通过 CLAUDE.md 注入项目知识，通过 memory 系统跨会话记忆用户偏好。

这个范式证明了一个关键洞察：**模型本身不是产品，模型外面的那层"壳"才是。** 裸模型只会说话。加上工具循环，它才能做事。加上上下文管理，它才能在特定场景里做事。加上记忆，它才能为特定的"你"做事。

但 Claude Code 已经如此优秀，为什么还要从头自己搓一个？

## 为什么要造轮子

三个理由。

**第一，模型绑定。** Claude Code 是 Anthropic 的产品，官方绑定 Claude 系列模型。你不能原装适配其他家的模型。虽然可以通过改配置文件或使用 CC-switch 等第三方工具切换，但毕竟是 hack，不是正统方案。如果你想让你的 Agent 自由奔跑在 GPT、Gemini、GLM、开源模型之间，你需要一个原生支持多模型的 Harness。

**第二，黑盒恐惧。** Claude Code 虽然出现过一次源码泄漏事件，但其项目过于庞大——约 57 万行代码。你日常使用中遇到问题时，很难深入到源码层面去理解和排查。你对它的信任建立在"它通常工作得很好"，而不是"我理解它为什么工作"。这种信任是脆弱的。

**第三，学习。** 这是最重要的理由。理解 Agent Harness 的最好方式，不是读文档，不是看视频，而是亲手造一个。造轮子的目的不是替代成品，而是理解原理。就像每个计算机专业的学生都要从零写一个操作系统内核——不是为了替代 Linux，而是为了理解操作系统的本质。

造完之后你会发现，Agent Harness 的核心比你想象的简单得多。真正复杂的部分——工具调用、上下文管理、记忆系统——每一个拆开来看都是可以理解的工程问题。而 LLM 作为"大脑"提供的推理能力，让这些工程组件的编排变得出乎意料地优雅。

## Harness 到底是什么

在动手之前，让我们把 Harness 这个概念拆得更清楚一些。

一个 Agent Harness 至少需要以下组件：

**工具循环（Tool Loop）** — 这是最核心的部分。一个 while 循环：接收用户输入 → 发给 LLM → LLM 返回响应（可能包含工具调用）→ 执行工具 → 把结果喂回 LLM → LLM 继续思考 → 循环，直到 LLM 认为任务完成。就这么简单。Claude Code 证明了工具循环是 Agent 的充分条件——不需要复杂的规划算法，不需要知识图谱，一个精心设计的工具循环加上一个足够强的模型，就能解决大部分实际问题。

**工具集（Tools）** — LLM 的"手脚"。文件读写、终端执行、代码搜索是最基础的。更高级的包括浏览器控制、数据库操作、API 调用、网络搜索。每一个工具本质上就是一个函数，Harness 负责把工具的定义告诉 LLM（以 JSON Schema 格式），然后根据 LLM 的调用请求执行对应的函数并返回结果。

**上下文管理（Context Management）** — LLM 的上下文窗口是有限的。Harness 需要聪明地管理什么信息放进上下文、什么信息压缩、什么信息丢弃。这包括系统提示（定义 Agent 的行为准则）、对话历史、工具返回结果、以及项目特定的知识注入。

**记忆系统（Memory）** — 跨会话的持久化存储。最简单的实现是文件系统——把重要信息写入文件，下次会话时读取。更复杂的实现包括向量数据库、全文搜索引擎、LLM 摘要。

**安全边界（Safety）** — 决定 Agent 能做什么、不能做什么。哪些命令可以执行，哪些文件可以访问，哪些网络请求可以发出。这是 Harness 的刹车系统。

当你把这五个组件组装在一起，你就得到了一个最小可用的 Agent Harness。

## 让 Claude Code 帮我们造

Agent Harness 是个不小的软件工程，即使是最小可行 MVP 版本，也有几千行代码。一个人从零写，可能需要几周甚至几个月。

但我们有一个秘密武器：让 Claude Code 来帮我们造。

这听起来有点套娃——用 Agent 来造 Agent。但这恰恰是这个时代最自然的工作方式。你不需要一行一行自己写代码，你需要的是：

1. **理解你要造什么**（上一节已经讲清楚了）
2. **找到优秀的蓝本参考**
3. **把控方向，让 CC 负责实现**

这里推荐两个优秀项目作为蓝本：

**Learn Claude Code**[^1] — 一个从零开始打造极简 Agent Harness 的教程。它的价值在于"教学"而不是"功能"——每一步都解释为什么这样做，代码量控制在最小。适合理解 Harness 的骨架结构。

**pi-agent**[^2] — 一个开源的、用 TypeScript 写的 Agent Harness。它更完整，包含了工具系统、上下文管理、多模型适配等实战级功能。适合作为"我要做一个能用的东西"的起点。

有了这两个蓝本，你的角色就从"程序员"变成了"架构师+产品经理"。你不需要写代码——让 CC 写。你需要做的是：

- 告诉 CC 你要实现什么功能
- 审查 CC 写出来的代码是否符合你的预期
- 测试功能是否正常
- 决定下一步该做什么

一个周末的时间，CC 就能帮你构建一个非常完整的项目。

## 核心架构：比你想的简单

让我用最直白的方式描述一个 Agent Harness 的核心循环，用 Python 伪代码：

```python
while True:
    # 1. 组装上下文：系统提示 + 对话历史 + 工具定义
    messages = build_context(system_prompt, conversation_history, tools)

    # 2. 调用 LLM
    response = llm.chat(messages)

    # 3. 检查 LLM 是否要调用工具
    if response.has_tool_calls:
        for tool_call in response.tool_calls:
            # 4. 执行工具
            result = execute_tool(tool_call.name, tool_call.arguments)
            # 5. 把结果喂回对话
            conversation_history.append(tool_call)
            conversation_history.append(result)
        # 继续循环，让 LLM 处理工具结果
        continue

    # 6. LLM 没有工具调用，说明它认为任务完成了
    print(response.text)
    break
```

这就是核心。大约 20 行代码。整个 Agent Harness 的"灵魂"就在这里。

剩下的一切——工具系统的实现细节、上下文窗口的管理策略、记忆的存储和检索、安全的边界设定——都是围绕这个核心循环的工程扩展。

理解了这个核心，你就能理解所有 Agent 产品的本质区别：Claude Code 的工具循环更精细（57 万行代码打磨），OpenClaw 加上了多通道 Gateway，Hermes Agent 在循环外层套了一个学习闭环。但它们的"心脏"都是这个 while 循环。

## 工具系统：给 Agent 装上手脚

核心循环之外，最重要的组件是工具系统。

一个工具的定义通常包含三部分：

```json
{
  "name": "read_file",
  "description": "读取指定路径的文件内容",
  "parameters": {
    "path": {"type": "string", "description": "文件的绝对路径"}
  }
}
```

LLM 看到这个定义后，就知道"我可以调用一个叫 read_file 的工具，需要提供一个 path 参数"。当它决定需要读文件时，会返回一个结构化的调用请求。Harness 收到请求后，执行对应的函数，把结果返回给 LLM。

一个最小可用的工具集通常包括：

- **文件操作**：读文件、写文件、列目录、搜索文件
- **终端执行**：运行 shell 命令并获取输出
- **网络请求**：HTTP GET/POST，用于调用外部 API

更高级的工具可以包括：浏览器控制（Puppeteer/Playwright）、数据库操作、代码解析（AST）、Git 操作。

工具系统的设计有一个关键原则：**每个工具做一件事，做到极致。** 不要做一个"万能工具"，而是做一组职责清晰的小工具。LLM 的推理能力会负责在合适的时机调用合适的工具。

## 上下文管理：Agent 的短期记忆

LLM 的上下文窗口是有限的（Claude 约 200K tokens，GPT-4 约 128K tokens）。 Harness 必须在这个窗口里塞入所有必要信息，同时不能浪费空间。

典型的上下文构成：

```
[系统提示] — 定义 Agent 的身份、行为准则、可用工具
[CLAUDE.md 等级的项目知识] — 注入项目特定的上下文
[对话历史] — 当前会话的完整对话记录
[工具调用和结果] — 历史工具调用的记录和返回值
[用户当前输入] — 最新一条消息
```

当对话变长时，上下文可能超出窗口限制。常见的处理策略：

- **滑动窗口**：只保留最近 N 轮对话
- **摘要压缩**：用 LLM 把早期对话压缩成摘要
- **关键信息提取**：只保留包含决策和结论的消息

Claude Code 用的是一种混合策略：context compressor 在后台工作，把冗长的对话浓缩成精华，为核心信息腾出空间。这个机制就像睡眠时的记忆巩固——你不需要记住昨天的每一秒，但你需要记住昨天学到了什么。

## 记忆系统：Agent 的长期记忆

对话历史是短期记忆，关掉就没了。真正的 Agent 需要长期记忆——跨会话的持久化存储。

最简单的实现是文件系统。Claude Code 就是这么做的：把用户的偏好、项目架构决策、工作习惯写成 Markdown 文件存在 `.claude/` 目录下。下次会话时读取这些文件，注入上下文。

更复杂的实现包括：

- **向量数据库**（如 Chroma、Pinecone）：存储文本的向量表示，支持语义搜索
- **全文搜索引擎**（如 FTS5）：支持关键词搜索对话历史
- **LLM 摘要**：定期把对话内容提炼成结构化的知识条目

Hermes Agent 的做法很有代表性：FTS5 全文搜索 + LLM 摘要的组合，让 Agent 能搜索自己过去所有对话记录。它不只是"记住"上次聊了什么，它能从历史经验中检索相关解决方案。这是从"记忆"到"经验"的跃迁。

## 安全边界：给 Agent 系上安全带

Agent 能读写文件、执行命令、访问网络——这些能力如果不加限制，就是安全隐患。

安全设计的基本原则：

**最小权限原则** — Agent 只应该拥有完成任务所需的最小权限。不需要写文件的场景，就不给写权限。

**白名单机制** — 明确列出允许执行的工具和命令，不在白名单内的一律拒绝。比黑名单更安全，因为你不可能穷举所有危险操作。

**沙箱隔离** — 在 Docker 容器或虚拟环境中执行不可信的操作。即使 Agent 被诱导执行了危险命令，影响范围也局限于沙箱内。

**审批流程** — 敏感操作（如删除文件、发送邮件、执行不可逆命令）需要用户确认。Claude Code 的 permission 系统就是这种设计。

安全的 Harness 不是"不信任 Agent"，而是"即使 Agent 犯错，也不会造成不可挽回的后果"。就像汽车的安全带不是因为不信任驾驶员，而是因为即使是最好的驾驶员也可能犯错。

## 从 MVP 到实战

经过一个周末的开发，CC 就帮你构建了一个完整的 Agent Harness。包含了工具循环、工具系统、上下文管理、记忆系统和安全边界——五大核心组件全部齐备。

我把它 MIT 开源[^3]了。

但开源不是终点。真正的学习发生在你使用它、调试它、改进它的过程中。

你会遇到上下文窗口不够用的问题——然后你就会深入理解上下文压缩策略。你会遇到工具调用出错的问题——然后你就会设计更健壮的错误处理。你会遇到 Agent "迷路"的问题——然后你就会思考如何通过更好的系统提示来引导 Agent 的行为。

这些问题的每一个，都是深入理解 Agent Harness 原理的入口。

## 造完轮子之后

造完自己的 Agent Harness 之后，你会获得一种"祛魅"的能力。

再看 Claude Code，你不再觉得它是魔法。你能看出它的工具循环在哪里、上下文管理用了什么策略、记忆系统是怎么实现的。你能理解它的 57 万行代码在做什么，因为你自己造的那个"轮子"包含了同样的核心逻辑。

再看 OpenClaw，你能理解它的 Gateway 架构是对多通道问题的优雅回答。再看 Hermes Agent，你能看懂它的学习闭环是如何在工具循环外层套上了一层经验积累系统。

造轮子的终极价值，是让你从"使用者"变成"理解者"。使用者按按钮，理解者知道按钮后面连着什么。

在 AI 时代，这种理解能力是最稀缺的。
