## 一个偶然的发现

今天我在测试 Hermes Agent（一个跑在我 MacBook 上的 AI 助手）时，发现了一个有趣的现象：

**我的电脑合盖超过 10 分钟后，通过微信给 Hermes 发消息，它居然还能回复！**

这让我很意外。按理说，MacBook 合盖应该进入睡眠状态，所有网络连接都会断开才对。于是我开始排查原因。

## 排查过程

首先查看系统的电源管理设置：

```bash
pmset -g
```

输出中一行引起了我的注意：

```
sleep  1 (sleep prevented by Electron, dasd, dasd)
```

**系统睡眠被阻止了！** 而且阻止它的正是 Electron 进程——也就是微信桌面客户端。

再查看具体的断言（assertion）：

```bash
pmset -g assertions
```

```
pid 17580(Electron): NoIdleSleepAssertion named: "Electron"
```

## 真相大白

原来，**微信桌面端（基于 Electron 框架）会向 macOS 注册一个「禁止空闲睡眠」的断言**。这意味着只要微信在运行，macOS 就不会因为空闲而进入睡眠。

再加上系统默认开启的两个辅助功能：

| 功能 | 作用 |
|------|------|
| **Power Nap（电能小憩）** | 睡眠期间仍可进行部分网络活动 |
| **tcpkeepalive** | 保持 TCP 长连接不断 |

三重保障下，MacBook 即使合盖，Hermes Agent 依然可以通过微信收发消息。

## 为什么这很实用？

对于像我这样在 MacBook 上自托管 AI 助手的用户来说，这个发现意味着：

> **不需要云服务器，只要插着电、开着微信，合上盖子就能走。**

相比部署到云上，省了一笔费用，而且本地运行的数据隐私也更有保障。

## 需要注意的限制

当然，这并不是完美的方案：

- 🔋 **电池模式**下 macOS 的电源管理更激进，可能会强制睡眠
- 🌐 **断网**时消息会丢失，不像云服务器有重连保障
- 🔄 **系统重启或更新**后需要手动启动 Hermes 和微信
- ⚡ 合盖不睡眠会**持续耗电**（虽然不多）

## 验证方法

如果你也想测试，可以这样做：

1. 打开终端，运行 `pmset -g` 查看是否有 `sleep prevented by Electron`
2. 合上 MacBook 盖子
3. 等待 10 分钟以上
4. 用手机给微信端的 AI 助手发消息，看是否能收到回复

## 结语

有时候最好的发现来自日常使用中的小疑问。一个简单的"为什么合盖了还能用？"就引出了对 macOS 电源管理机制的深入了解。

如果你也在 MacBook 上跑着什么需要持续在线的服务，不妨检查一下你的 `pmset -g` 输出，也许会有意想不到的发现。
